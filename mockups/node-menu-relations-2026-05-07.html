업무 노드 클릭 시 나타나는 메뉴 레이아웃 비교 · A안 (통합) vs B안 (분리)

★ 추천 A안 모태에 KPI 통합 (2 섹션)

KPI를 "모태"의 한 종류로 흡수. 흐름도에서 KPI는 어차피 task의 위에서 화살표로 오는 부모 노드라 의미 일치.

B안 3 섹션 분리 (KPI 별도)

"연계"를 KPI 전용으로 분리. 카테고리 구분이 명확하지만 메뉴가 세로로 길어지고 흐름도와 의미가 어긋남.

비교 항목 A안 (통합 2 섹션) B안 (분리 3 섹션)
메뉴 세로 길이 짧음 — 섹션 2개 길어짐 — 섹션 3개로 스크롤 필요할 수도
흐름도 의미 일치 일치 — KPI도 화면상 task의 부모로 보임 어긋남 — KPI는 별도 카테고리이지만 차트에선 부모처럼 보임
KPI / 업무 시각 구분 chip 색상으로 구분 (초록=KPI, 파랑=업무) 섹션 자체가 분리되어 명확
"빈 상태"에서 KPI 연결 자연스러움 — 모태 섹션 한 곳에서 KPI/업무 둘 다 추가 연계 섹션을 따로 찾아야 함
사용자 학습 비용 "모태에 KPI 포함" 컨셉 한 번 익히면 됨 3 카테고리 차이를 매번 떠올려야 함
한 번에 여러 종류 처리 한 dropdown에서 KPI + 업무 같이 검색 섹션마다 따로 dropdown
구현 난이도 적음 — dropdown에 optgroup 추가 중간 — 새 섹션 + KPI 전용 핸들러 분리
최종 추천 — A안
① 메뉴가 컴팩트해서 클릭 부담이 적고, ② 흐름도(KPI가 업무의 윗쪽 화살표 시작점)와 의미가 일치하며, ③ 한 dropdown에서 KPI/업무 모두 검색 가능해서 빠릅니다. 단점인 "KPI ↔ 업무 혼동"은 chip 색상(초록=KPI, 파랑=업무)과 아이콘(▣ KPI, ◆ 업무)으로 충분히 해결 가능. B안은 "KPI 카테고리를 명확히 분리해야 한다"는 정책적 이유가 있을 때만 선택.
